Graph

Run agents as nodes in a directed acyclic graph (DAG). Independent nodes execute in parallel; dependent nodes wait for their inputs automatically.

When to Use Graph

  • You know all the steps before execution starts
  • Some steps are independent and can run concurrently
  • You want predictable, auditable execution order
  • Downstream agents need the results of upstream agents as context

How It Works

Each node is an agent with an ID and an optional dependencies list. Nodes with no unsatisfied dependencies run immediately; nodes with dependencies wait until all their inputs are ready. The orchestrator handles scheduling automatically.

Basic Example

A CI/CD-style pipeline: lint and test run concurrently. Once both pass, the build agent runs using their results as context.

ci_pipeline.swiftSwift
import StrandsAgents

let lintAgent = Agent(
    model: provider,
    tools: [readSourceFiles],
    systemPrompt: "Analyze code for style violations and errors. Report each issue with file and line."
)

let testAgent = Agent(
    model: provider,
    tools: [runTestSuite],
    systemPrompt: "Run the test suite. Report any failures with their error messages."
)

let buildAgent = Agent(
    model: provider,
    tools: [compileBuild],
    systemPrompt: "Build the project. You will receive lint and test results as context first."
)

let graph = GraphOrchestrator(nodes: [
    GraphNode(id: "lint",  agent: lintAgent),
    GraphNode(id: "test",  agent: testAgent),
    // build waits for both lint and test to finish
    GraphNode(id: "build", agent: buildAgent, dependencies: ["lint", "test"]),
])

let result = try await graph.run("Run CI checks on the latest commit")
print(result.output)  // build agent's final summary

Chaining Stages

Stages chain by declaring dependencies. Each node's output is forwarded as context to every node that lists it as a dependency — no manual wiring needed.

A full CI/CD pipeline shows this clearly: three checks run in parallel, then each subsequent stage unblocks only after the previous one completes.

Full pipelineSwift
let graph = GraphOrchestrator(nodes: [
    // Stage 1 — three independent checks run concurrently
    GraphNode(id: "lint",     agent: lintAgent),
    GraphNode(id: "test",     agent: testAgent),
    GraphNode(id: "security", agent: securityAgent),

    // Stage 2 — build waits for all three
    GraphNode(id: "build",  agent: buildAgent,  dependencies: ["lint", "test", "security"]),

    // Stage 3 — deploy waits for a clean build
    GraphNode(id: "deploy", agent: deployAgent, dependencies: ["build"]),

    // Stage 4 — smoke test waits for a live deployment
    GraphNode(id: "smoke",  agent: smokeAgent,  dependencies: ["deploy"]),
])

let result = try await graph.run("Release v2.1.0")
print(result.output)

Parallel Execution

The orchestrator resolves the dependency graph at runtime. In the pipeline above, lint, test, and security all start at the same moment — they don't wait for each other. This is the main performance benefit over running agents sequentially.

💡

Nodes are declared in any order — the dependency list, not the declaration order, determines execution. Add as many independent nodes as you like; they all run concurrently.

Accessing Intermediate Results

GraphResult contains the output of every node, keyed by ID. Useful for logging, debugging, or surfacing intermediate results in your UI:

Swift
let result = try await graph.run("Release v2.1.0")

print(result.output)  // final node's output (smoke test summary)

// Inspect any intermediate node
print(result.nodeOutputs["lint"]     ?? "")  // lint report
print(result.nodeOutputs["security"] ?? "")  // security findings
print(result.nodeOutputs["build"]    ?? "")  // build log
